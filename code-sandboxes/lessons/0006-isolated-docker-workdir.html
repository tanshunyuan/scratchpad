Isolated Docker workdir

Mission: make local Docker a better sandbox bridge before future remote runtime.

Tangible win: after this lesson, you should know why writable bind mounts are weak sandboxing, and how to run preview from a disposable copy.

Problem with current Docker shape

This command mounts the real workspace as the app directory:

docker run --rm \
  -v "$PWD:/app" \
  -w /app \
  -p 5174:5174 \
  node:24 \
  sh -lc "pnpm install && pnpm dev --host 0.0.0.0 --port 5174"

It isolates process/dependencies, but not project writes.

container writes /app/src/App.tsx
→ host file changes too

So local Docker is a bridge, not the final sandbox. Source: Docker bind mount docs.

Better local shape

Run the app from a disposable copy, not the real workspace.

real workspace
→ copy to temp preview dir
→ mount temp preview dir into container
→ run dev server
→ discard temp dir on stop/restart

Now bad app writes hit the temp copy, not the real workspace.

Manual version

From code-examples/sample-preview-app:

PREVIEW_DIR="/tmp/sample-preview-run"
rm -rf "$PREVIEW_DIR"
mkdir -p "$PREVIEW_DIR"
rsync -a --delete \
  --exclude node_modules \
  --exclude .pnpm-store \
  ./ "$PREVIEW_DIR/"

docker run --rm \
  -v "$PREVIEW_DIR:/app" \
  -v sample-preview-node-modules:/app/node_modules \
  -w /app \
  -p 5174:5174 \
  node:24 \
  sh -lc "corepack enable && pnpm install && pnpm dev --host 0.0.0.0 --port 5174"

What improves

  • Container can mutate /app without touching original workspace.
  • Preview run becomes disposable.
  • Closer to remote sandbox mental model: upload/copy files, run, discard.
  • Electron still gets same iframe URL.

What still is not solved

  • User still needs Docker daemon running.
  • Container may still access network unless restricted.
  • Initial file copy can become slow for large projects.
  • Changes made by the running app do not sync back unless you explicitly choose to.

Future remote sandbox shape

This local copy model maps cleanly to remote later:

local workspace
→ package/sync files
→ remote container workdir
→ remote dev server
→ preview URL/proxy
→ logs stream back

Remote sandbox is bigger win because user no longer needs local Docker, and runtime can be centrally controlled. But it adds backend infra.
