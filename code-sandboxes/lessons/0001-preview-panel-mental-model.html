Preview panel mental model

Mission: add live right-side preview to your Electron coding app, with simple infra.

Tangible win: after this lesson, you should know what component owns editor, runtime, preview, env, and logs.

One idea

The preview panel is not magic. It is a browser surface loading a real app server.

Electron app
├─ React editor UI
├─ Electron main process
│  └─ starts dev server with Node child_process
└─ Preview pane
   └─ loads http://localhost:5173

For local Electron MVP, use a normal dev server like Vite. Vite handles hot module replacement, so saved files update preview quickly. Source: Vite HMR docs.

Who does what?

  • Renderer React app: file tree, editor, env UI, log UI.
  • Electron main: trusted side. Starts/stops app process, reads/writes files, stores env.
  • Dev server: runs user app, usually npm run dev.
  • Preview pane: loads dev server URL using WebContentsView or iframe. Source: Electron WebContentsView docs.

Smallest working flow

1. User opens project folder.
2. Electron main runs: npm install if needed.
3. Electron main runs: npm run dev.
4. Main waits until port is ready.
5. Preview loads http://localhost:PORT.
6. Editor saves file.
7. Vite HMR refreshes preview.

Where env fits

Env panel stores key/value pairs. Electron main injects them when spawning dev server. Node supports this with child_process.spawn(). Source: Node child_process docs.

spawn('npm', ['run', 'dev'], {
  cwd: projectDir,
  env: {
    ...process.env,
    ...projectEnv
  }
})

When Docker becomes useful

Skip Docker for first MVP if code is yours/trusted. Add Docker when code becomes untrusted, dependencies conflict, or you need reproducible isolation. Source: Docker container docs.

Electron → local Node controller → Docker container → port mapping → preview

Security rule

Preview may run arbitrary user JS. Keep it isolated. Do not give preview Node access. Follow Electron security guidance: Electron Security Tutorial.
