Preview error feedback loop

Mission: when agent edits templated workspace files, preview should show outcome and expose errors clearly.

Tangible win: after this lesson, you should separate “server died” from “app has compile/runtime error”.

Where you are now

Your runner lifecycle is good enough:

start → spawn dev server → wait for URL → running
stop → SIGTERM → wait exit → stopped
restart → stop then start

Next important loop is feedback after file changes. In target app, the agent usually changes files, not the user typing manually.

One idea

A preview can fail while the dev server is still healthy.

agent edits file
→ Vite watches filesystem
→ rebuild/HMR runs
→ app either updates or errors
→ iframe shows app/overlay
→ logs show what happened

Vite watches files and uses HMR for updates. Source: Vite HMR docs.

Do not collapse all failures into “server down”

These are different:

server not running       → no process / port unavailable
server starting          → process exists, URL not ready yet
server running           → URL responds
app compile error        → server alive, app failed to build
app runtime error        → server alive, browser JS crashed
iframe load error        → browser could not load URL

Your current status mostly tracks server lifecycle. That is fine. App errors can be shown through logs and iframe overlay without changing server status to stopped.

Why logs matter

Logs are not only debug noise. They are the feedback channel for agent work.

[stdout] Vite ready on http://localhost:5174
[stderr] Failed to resolve import "./Missing"
[stderr] SyntaxError: Unexpected token

When agent makes a bad edit, logs tell what to fix. The iframe may also show Vite error overlay.

Good mental model

runner health: is dev server alive?
app health: did current code build/run?
preview surface: what does iframe show?

Keep these separate. One status field can be simple for now, but your brain should track three layers.

Small test

Use your current sample app.

1. Start preview.
2. Edit sample-preview-app/src/App.tsx.
3. Add a syntax error.
4. Watch iframe and logs.
5. Fix syntax error.
6. Confirm preview recovers without restart.

If preview recovers after fixing file, your feedback loop works.

What not to build yet

  • No custom error parser.
  • No diagnostics panel.
  • No project command detection.
  • No Docker.

For MVP, raw logs + iframe overlay are enough.

Retrieval check
