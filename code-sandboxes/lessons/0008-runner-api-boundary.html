Runner API boundary

Mission: make Electron talk to a runner API, so Docker/runtime can move remote later.

Tangible win: after this lesson, you should know what logic belongs in Electron and what belongs in the runner service.

Where you are now

You created a runner skeleton at code-examples/remote-preview-runner/ with these intended routes:

POST /sandboxes/start
POST /sandboxes/:id/stop
GET  /sandboxes/:id/logs

This is right direction. The runner API is the seam between local UI and sandbox runtime.

One idea

Electron should not know Docker details.

Before:
Electron → spawn docker run → container → preview URL

After:
Electron → runner API → docker/container → preview URL

The runner owns sandbox lifecycle. Electron owns UI, iframe, buttons, and showing logs/status.

Why this boundary matters

Today runner can be local Express server. Later it can be remote service. Electron code should barely change.

local runner URL:  http://localhost:8086
remote runner URL: https://runner.example.com

Same API shape. Different host.

Responsibility split

Electron app
- start/stop/restart buttons
- iframe loads previewUrl
- displays logs/status
- sends workspace snapshot or workspace ID

Runner service
- receives start/stop requests
- creates sandbox ID
- starts container/dev server
- knows ports/container IDs
- streams stdout/stderr/status
- cleans up resources

Minimal API contract

POST /sandboxes/start
request:
{
  "workspacePath": "/path/to/workspace"
}

response:
{
  "sandboxId": "sandbox_123",
  "previewUrl": "http://localhost:5174"
}
GET /sandboxes/sandbox_123/logs
response: text/event-stream

event: status
data: {"status":"starting"}

event: log
data: {"type":"stdout","text":"Vite ready"}
POST /sandboxes/sandbox_123/stop
response:
{
  "ok": true
}

Server-Sent Events are simple one-way streaming from runner to Electron. Source: MDN Server-Sent Events. Express route handling source: Express routing docs.

State moves into runner

Electron currently has devServerProcess. With runner boundary, runner owns this state.

const sandboxes = new Map()

sandboxes.set(sandboxId, {
  process,
  previewUrl,
  status
})

Electron stores only:

{ sandboxId, previewUrl, status, logs }

Preview URL stays boring

This is the win from iframe design.

Electron receives previewUrl
→ iframe.src = previewUrl

Local Docker, local runner, remote runner: all become URL providers.

Retrieval check

Question: after adding runner API, who should know Docker command details?

Next tiny build step

In remote-preview-runner, implement only POST /sandboxes/start first.

POST /sandboxes/start
→ spawn Docker using existing command
→ wait for preview URL
→ return { sandboxId, previewUrl }

Do not implement auth, upload, proxy, or multi-user cleanup yet. Keep runner local. Goal is boundary, not cloud.
