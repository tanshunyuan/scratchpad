Preview runner lifecycle

Mission: make current preview runner easier to reason about. No env vars. No project picker. Keep iframe.

Tangible win: after this lesson, you should know why logs are not readiness, and what states preview runner needs.

Where you are now

Your current MVP already has the core runtime loop.

renderer button
→ window.api.startPreview()
→ ipcMain.handle('preview-start')
→ spawn('pnpm', ['dev', '--port', '5174'])
→ stdout/stderr logs stream to renderer
→ iframe loads http://localhost:5174
→ stop sends SIGTERM

This means the next lesson is not “build runner from scratch”. It is “make runner lifecycle correct”.

One idea

Process started and preview ready are different events.

spawn event      = OS created process
stdout data      = process printed text
port responds    = browser can load preview
exit event       = process is dead

A Vite server can print logs before it is fully ready. Or it can print an error and never become reachable. So stdout should create log lines, not prove preview readiness.

Useful states

stopped  → no child process
starting → child process exists, port not ready yet
running  → port responds
stopping → SIGTERM sent, waiting for exit
error    → process failed or exited during startup

These states are enough for MVP. Do not overbuild state machine abstractions yet.

Better start flow

1. If process already exists, do nothing.
2. Set status = starting.
3. Spawn dev server.
4. Stream stdout/stderr to logs.
5. Poll http://localhost:5174 until it responds.
6. Set status = running.
7. Let iframe load/reload preview URL.

Better stop flow

1. If no process exists, status = stopped.
2. Set status = stopping.
3. Send SIGTERM.
4. Wait for exit event.
5. Set status = stopped.
6. Clear process reference.

SIGTERM asks process to shut down. exit confirms it actually shut down.

Restart is stop + start

Restart should not be magical.

restart
→ stop current process
→ wait for exit
→ start new process
→ wait for port ready
→ reload iframe

This matters when dependencies changed, dev server wedged, or preview needs clean reset.

Iframe constraint

Keep iframe. It is portable if future app is not Electron-only.

Electron today: iframe loads local URL
Future non-Electron app: iframe still loads preview URL

Electron main is current runner host, but preview surface does not need to become Electron-specific.

Retrieval check

Question: what proves preview is ready for iframe?

Next tiny build step

Add a small waitForUrl() helper in Electron main. Use it before setting status to running.

async function waitForUrl(url: string, timeoutMs = 15000) {
  const startedAt = Date.now()

  while (Date.now() - startedAt < timeoutMs) {
    try {
      const response = await fetch(url)
      if (response.ok) return
    } catch {
      // server not ready yet
    }

    await new Promise((resolve) => setTimeout(resolve, 250))
  }

  throw new Error(`Timed out waiting for ${url}`)
}

After this, status means more:

starting = process launched, waiting for app
running  = iframe can load app
