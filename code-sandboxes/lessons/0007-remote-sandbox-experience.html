Remote sandbox experience

Mission: preview code in a remote runtime while Electron stays local UI.

Tangible win: after this lesson, you should name the pieces needed for remote preview: sync, runner, logs, preview URL, cleanup.

Where local Docker got us

Local Docker taught the shape:

workspace files
→ isolated runtime
→ dev server
→ preview URL
→ logs/status back to app

But local Docker still depends on user machine and local Docker daemon. Remote sandbox moves runtime elsewhere.

One idea

Remote sandbox is local Docker split across network.

Electron app
→ upload/sync workspace
→ remote runner starts container
→ container runs dev server
→ service exposes preview URL
→ Electron iframe loads preview URL
→ logs stream back

Core pieces

1. Workspace sync
   Local files become remote workdir.

2. Runner API
   Start, stop, restart sandbox.

3. Remote container
   Installs deps and runs dev server.

4. Preview routing
   Public/authorized URL points to container port.

5. Log/status stream
   stdout/stderr and lifecycle events return to Electron.

6. Cleanup policy
   Stop container, delete workdir, expire idle sandboxes.

Electron barely changes

Your current iframe choice still works.

local Docker iframe URL:  http://localhost:5174
remote iframe URL:        https://preview-abc.example.com

The preview surface remains URL-based. That was the right portability constraint.

Runner API shape

POST /sandboxes
→ creates sandbox
→ returns sandboxId

POST /sandboxes/:id/files
→ uploads workspace snapshot or patch

POST /sandboxes/:id/start
→ starts container/dev server
→ returns previewUrl

GET /sandboxes/:id/logs
→ streams stdout/stderr/status

POST /sandboxes/:id/stop
→ stops container

Logs can stream with WebSocket or Server-Sent Events. Sources: MDN WebSocket API, MDN Server-Sent Events.

Preview routing model

Remote container listens on an internal port. A proxy maps external URL to that container.

https://preview-abc.example.com
→ preview proxy
→ sandbox container :5174

This is remote version of Docker -p host:container.

What this unlocks

  • User does not need Docker locally.
  • Runtime is controlled by your service.
  • Untrusted code runs away from user machine.
  • Sandbox can be killed/discarded centrally.

What it costs

  • Backend runner service.
  • File upload/sync protocol.
  • Preview proxy/routing.
  • Auth and access control.
  • Resource limits, timeouts, cleanup jobs.

Retrieval check

Question: in remote sandbox, what replaces localhost:5174?

Next tiny design step

Sketch minimal remote runner contract, not full infra:

startSandbox(workspaceSnapshot) → { sandboxId, previewUrl }
stopSandbox(sandboxId) → { ok }
streamLogs(sandboxId) → stdout/stderr/status events

This is enough to map your current Electron runner to future remote runner.
