Docker sandbox mental model

Mission: Electron starts isolated preview runtime, iframe still loads a local URL.

Tangible win: after this lesson, you should explain how Docker bridges local app, workspace files, and preview URL.

Where you are now

Your current runner starts Vite directly on your machine:

Electron main
→ spawn('pnpm', ['dev'])
→ Vite runs on host
→ iframe loads http://localhost:5174

That works. Docker changes where the dev server runs, not what iframe loads.

One idea

Docker adds an isolation boundary between Electron and app code.

Electron main
→ spawn('docker', ['run', ...])
→ container runs pnpm dev
→ host maps container port 5174
→ iframe loads http://localhost:5174

Electron still controls start/stop/logs. The app code runs inside container.

The four bridges

1. Filesystem bridge: host workspace → container /app
2. Working-dir bridge: run command from /app
3. Port bridge: container 5174 → host 5174
4. Log bridge: docker stdout/stderr → Electron logs

Docker docs call these bind mounts and published ports. Sources: Docker bind mounts, Docker publishing ports.

Command shape

docker run --rm \
  -v /host/workspace:/app \
  -w /app \
  -p 5174:5174 \
  node:22 \
  sh -lc "corepack enable && pnpm install && pnpm dev --host 0.0.0.0 --port 5174"

Meaning:

--rm                 remove container after exit
-v host:/app         make workspace visible inside container
-w /app              run command from workspace
-p 5174:5174         expose container preview to host
node:22              base runtime image
sh -lc "..."         run install + dev command inside container

Important Vite detail

Inside Docker, dev server must listen on all container interfaces:

pnpm dev --host 0.0.0.0 --port 5174

If it only listens on container localhost, host port mapping may not reach it. Vite documents --host for exposing server externally. Source: Vite CLI docs.

What improves

  • Dependencies install inside container, not host app environment.
  • Preview process is easier to delete as container.
  • Runtime becomes closer to remote/sandbox model.
  • Electron still gets simple localhost preview URL.

What Docker does not magically solve

  • Bind-mounted files can still be changed by container.
  • Network may still be available unless restricted.
  • Secrets must still be handled carefully.
  • Containers are isolation, not perfect security.

Next tiny build step

Before wiring Electron, run Docker manually against the sample app.

cd code-examples/sample-preview-app

docker run --rm \
  -v "$PWD:/app" \
  -w /app \
  -p 5174:5174 \
  node:22 \
  sh -lc "corepack enable && pnpm install && pnpm dev --host 0.0.0.0 --port 5174"

Then open:

http://localhost:5174

If browser shows app, Docker bridge works. Electron integration comes after.
