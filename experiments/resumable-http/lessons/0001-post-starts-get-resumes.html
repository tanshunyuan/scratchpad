POST starts. GET resumes.

Tiny mental model for OpenAI/Flue/AI SDK-style resumable streams in TypeScript.

Mission tie-in

You want a page refresh to behave like a lost listener, not like canceled work. Today’s win: name the three moving parts so framework docs stop feeling magical.

The trick

1. POST starts
POST /runs
Creates run, starts producer, streams first response.
→
2. Server records
Each chunk gets eventId and is appended to a log.
3. Refresh drops client
HTTP connection closes. Producer keeps running.
→
4. GET resumes
GET /runs/:id/events?after=17
Replay missed chunks, then follow live.

AI SDK documents this as two endpoints: POST creates a stream; GET checks/resumes an active stream for a chat ID [AI SDK]. Flue exposes a similar split: POST sends input to an agent instance; GET provides event streaming [Flue].

Why event ID, not byte offset?

AI streams are semantic events: token deltas, tool calls, message starts/ends. Byte offsets are brittle once encoding, JSON framing, or protocol versions change. SSE already has id:; browsers define Last-Event-ID reconnect behavior around it [MDN].

Minimal Node shape

type Event = { id: number; data: unknown };
type Run = {
  nextId: number;
  events: Event[];
  clients: Set<ServerResponse>;
  done: boolean;
};

const runs = new Map<string, Run>();

function emit(run: Run, data: unknown) {
  const event = { id: run.nextId++, data };
  run.events.push(event);

  for (const res of run.clients) {
    res.write(`id: ${event.id}\n`);
    res.write(`data: ${JSON.stringify(event.data)}\n\n`);
  }
}

function attach(req: IncomingMessage, res: ServerResponse, run: Run, after = 0) {
  res.writeHead(200, {
    "content-type": "text/event-stream",
    "cache-control": "no-cache",
    connection: "keep-alive",
  });

  for (const event of run.events) {
    if (event.id > after) {
      res.write(`id: ${event.id}\n`);
      res.write(`data: ${JSON.stringify(event.data)}\n\n`);
    }
  }

  if (run.done) return res.end();

  run.clients.add(res);
  req.on("close", () => run.clients.delete(res));
}

Important: req.on("close") removes that listener only. It must not cancel the producer. AI SDK calls this out: disconnects and refreshes should not be treated as server-side cancellation [AI SDK troubleshooting].
