Send runId as a metadata event

Your server now uses crypto.randomUUID(). The client needs that ID before it can resume.

Goal

Keep the stream as the only response body. The first event tells the browser which run was created.

id: 0
data: {"type":"run","runId":"..."}

id: 1
data: {"type":"delta","delta":"token-1"}

Why metadata event?

POST already returns a long-lived stream. Instead of returning separate JSON first, put control info inside the stream protocol. This is how richer AI stream protocols work: not every event is a token.

Step 1 — widen event data

Change your current StreamEvent type from “only delta” to a union:

type StreamEvent = {
  id: number;
  data:
    | { type: "run"; runId: string }
    | { type: "delta"; delta: string };
};

Step 2 — producer emits typed deltas

Your startProducer loop currently emits { delta: ... }. Make event type explicit:

emit(run, { type: "delta", delta: `token-${id}` });

Step 3 — POST writes run metadata once

After attach, write one metadata event directly to this POST response:

attach(req, res, run, 0);

writeEvent(res, {
  id: 0,
  data: { type: "run", runId: run.id },
});

runs.set(run.id, run);
startProducer(run);
Important: use writeEvent, not emit. Metadata should go only to the first POST response. It should not become part of replayed token history.

Step 4 — test with curl

curl -N -X POST http://localhost:8086/runs

Copy the runId, stop curl after a few tokens, then resume:

curl -N "http://localhost:8086/runs/<runId>/events?after=3"

Expected: resumed GET does not replay id: 0. It replays token events after 3, then follows live.

Retrieval check

  1. Why does the metadata event use writeEvent instead of emit?
  2. Why is metadata event id: 0 not saved in run.events?
  3. After refresh, what two values does the browser need to resume?
