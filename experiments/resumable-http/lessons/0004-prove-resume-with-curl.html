Prove resume with curl

Before browser UI, prove the contract: disconnect, reconnect, replay missed events.

What your current code has

Your code now has the core resumable shape: Run owns the replay log, listeners are live HTTP responses, POST /runs starts the producer, and GET /runs/1234/events?after=N only attaches to the existing run.

Next win is not more architecture. Prove the behavior with curl.

Test 1 — start then kill listener

# terminal 1
cd code-examples/tiny-typescript-demo
pnpm dev

# terminal 2
curl -N -X POST http://localhost:8086/runs

After event 3, press Ctrl-C. That kills listener A. Producer should continue in server process.

Test 2 — resume same run

# terminal 2 again
curl -N "http://localhost:8086/runs/1234/events?after=3"

Expected: first replay 4, 5, maybe more, then continue live until 10.

Test 3 — resume after complete

curl -N "http://localhost:8086/runs/1234/events?after=6"

Expected: replay 7..10, then connection closes because run.done === true.

Keep these shortcuts for now

Hardcoded 1234 is fine. It makes curl testing easy. Later replace it with crypto.randomUUID() and send the ID to the client.

In-memory runs is fine. It proves the model. Later Redis/DB replaces it so refresh survives process restart and multiple servers.
