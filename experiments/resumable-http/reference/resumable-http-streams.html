Resumable HTTP Streams

Core recipe for AI-style streaming in TypeScript.

One-sentence model

POST starts work; server keeps work alive; GET resumes by replaying missed event IDs, then follows live output.

Pieces

runId / chatId
Stable ID for current generation/session.
eventId
Monotonic cursor: 1, 2, 3… for chunks/events.
event log
Buffer/persist emitted chunks so reconnect can replay.
live subscribers
Open HTTP responses currently receiving new chunks.

Flow

POST /runs
  create runId
  start producer in background
  stream events to this first client

GET /runs/:runId/events?after=17
  replay events where id > 17
  keep connection open
  send future events live

SSE event shape

id: 18
event: chunk
data: {"delta":"hello"}

Use id: as resume cursor. With EventSource, browsers can send Last-Event-ID on reconnect; with fetch() streaming, store and send the cursor yourself.

Rules

  • Do not use byte offsets for AI/UI streams. Use event IDs.
  • Do not cancel producer on page refresh. Treat disconnect as “no listener”, not “stop work”.
  • Persist event log in Redis/DB for production. Memory only works for demos and one process.
  • Separate stop from disconnect: explicit stop endpoint cancels producer; refresh does not.

Framework mapping

  • AI SDK: POST creates stream; GET /api/chat/[id]/stream resumes active stream.
  • Flue: POST sends input to an agent instance; GET exposes event streaming.
  • OpenAI-style APIs: provider streams current request; your app must persist/replay if browser resume is required.
