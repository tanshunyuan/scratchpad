Resumable HTTP Streams

Reference for the Node + TypeScript demo and AI SDK-style mental model.

Core model

POST starts work. Server keeps producing even if listener disconnects. GET resumes by replaying missed event IDs, then follows live output.

connect → read chunks → parse SSE events → update state → render
refresh → reconnect with runId + lastEventId → replay missed → continue live

Server pieces

run.id
Stable ID for one generation. Browser needs it to resume.
nextEventId
Server-owned monotonic counter: 1, 2, 3…
events[]
Replay log. Never mutate it during resume.
listeners
Set of live ServerResponse objects. Zero, one, or many.
done
Producer finished. Replay remaining events, then close.
cancelled
User explicitly stopped work. Producer should stop.

Endpoint shape

POST /runs
  create run
  attach POST response as first listener
  send metadata event: { type:"run", runId }
  start producer

GET /runs/:runId/events?after=N
  find run
  replay events where id > N
  if done: close
  else: attach GET response as live listener

POST /runs/:runId/stop
  cancelled = true
  done = true
  close listeners

SSE framing

id: 1
data: {"type":"delta","delta":"token-1"}

id: 2
data: {"type":"delta","delta":"token-2"}
  • \n separates fields inside one event.
  • \n\n ends one event.
  • id: is the resume cursor.
  • data: is app payload, often JSON.

Producer vs listener

ThingMeaningCan disappear?
producerCode generating tokens/events over time.No, not on refresh.
listenerOne open HTTP response/socket receiving events.Yes, on refresh/network drop.
event logSaved emitted events used for replay.No, until expiry/cleanup.

Important functions

writeEvent(res, event)

Low-level send existing event to one response.

res.write(`id: ${event.id}\n`);
res.write(`data: ${JSON.stringify(event.data)}\n\n`);

emit(run, data)

Producer creates new event, stores it, broadcasts to current listeners.

const event = { id: run.nextEventId++, data };
run.events.push(event);
for (const listener of run.listeners) writeEvent(listener, event);

attach(req, res, run, after)

Connect one HTTP response to one run.

write SSE headers
for each event in run.events:
  if event.id > after: writeEvent(res, event)
if run.done: res.end()
else:
  run.listeners.add(res)
  req.on("close", () => run.listeners.delete(res))

Browser client loop

Start:
  POST /runs
  read response.body as ReadableStream
  parse SSE blocks
  type:"run"   → localStorage.runId = runId
  type:"delta" → localStorage.lastEventId = id; render delta

Page load:
  if runId exists:
    GET /runs/:runId/events?after=lastEventId

SSE parser mental model

fetch() gives arbitrary byte chunks. A chunk can contain half an event, one event, many events, or many events plus a partial tail.

let pendingText = "";

function parseEvents(text) {
  pendingText += text;

  const blocks = pendingText.split("\n\n");
  pendingText = blocks.pop() ?? ""; // unfinished tail, or "" if clean boundary

  return blocks.map(parseEventBlock);
}
"id: 1\ndata: A\n\n".split("\n\n")
// ["id: 1\ndata: A", ""]
// pendingText = ""; parse event 1

"id: 3\ndata:".split("\n\n")
// ["id: 3\ndata:"]
// pendingText = "id: 3\ndata:"; parse nothing yet

"id:1\ndata:A\n\nid:2\ndata:B\n\nid:3\ndata:".split("\n\n")
// parse event 1 + 2; keep "id:3\ndata:" pending

Production gaps

  • Memory dies on server restart. Use Redis/DB for real apps.
  • One Node process only. Multi-server needs shared stream storage/pub-sub.
  • Add auth: user must be allowed to resume/stop that run/chat.
  • Add cleanup/expiry for old runs and event logs.
  • Add error events if producer fails.

Mission recap

  • POST can start because it carries user input and opens first stream.
  • GET usually resumes because reconnecting is idempotent: “show me active stream for this ID.”
  • Resume needs two IDs: stream/run ID and last seen event ID.
  • Replay log bridges time when no listener is connected.
